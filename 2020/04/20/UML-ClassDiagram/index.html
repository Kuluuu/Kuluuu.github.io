UML学习笔记——类图

类图

UML中的类图的相关知识。

类图的概念

●类图描述系统中类的静态结构，描述系统中的类及类之间的关系，还描述类的内部结构(类的属性和操作)。

●类图是面向对象系统建模中最常用、最重要的图。

●类图是定义其它图的类图，在类图的基础上，可以使用状态图、协作图、顺序图、组件图等进一步描述系统其它方面的特征。

●对于抽象建模和设计实际的程序而言，类图都是有用的。

●类图中的类可以直接用某种面向对象编程语言实现。C++、java、C#...

类图的元素

①类(Class)

②接口( Interface )

③依赖关系(Dependency)

④泛化关系(Generalization)

⑤关联关系(Association)

⑥实现关系(Realization)

⑦聚合关系(aggregation)

⑧组合关系( composition)

类

类是构成类图的基础，也是面向对象系统组织结构的核心。

类是对一组具有相同属性、操作、关系和语义的对象的抽象。

类包括名称部分(Name)、属性部分(Attribute)和操作部分(Operation)

  • UML规范采用一个具有3个分栏的图标表示一个类。

属性

属性可见性(访问权限，可访问性)

可见性描述了该属性对其它类是否可见，以及是否可以被其它类引用。

类型：

①公有(Public) “+”

②私有( Private ) “-”

③受保护( Protected) “#”

注意：UML未规定默认的可见性，如果没有给出，表明该属性的可见性尚未定义。

作用域描述

作用域(scope)是与属性和操作相关的一一个重要概念。

  • 存在两种作用域:

(1)实例(instance) 作用域下，类的不同实例对象拥有自己的属性值，类

似于C++中的非静态成员。

(2)分类(classifier) 作用域下，类的所有实例对象共享相同的属性值，类，

似于C++中的静态成员。

  • UML分类作用域的属性和操作名字要带下划线。

抽象类

抽象类：至少拥有一个抽象操作的类。

抽象操作：在指定该操作的类中并没有该操作的实现方法，而只是

显示了操作签名的那种操作。例如，C++中的纯虚函数，Java中的抽象方法

在UML图中，抽象操作的显示是在操作签名后面用特性字符串，{abstract}，或者将操作签名用斜体字体方式显示。

在UML图中，抽象类的显示是在类名后面用特性字符串{abstract},或者将类名用斜体字体方式显示。

接口

  • 接口是一组没有相应实现方法的描述。

  • 接口通常被描述抽象操作，即用操作名、参数表和返回类型说明接口行为，而操作的实现部分将出现在实现该接口的元素中。

  • 使用接口可以很好地将类所需要的行为与该行为如何被实现完全分开。

  • 在进行系统建模时，接口起到十分重要的作用，因为模型元素之间的协作是通过接口进行的。

  • UML中接口通常只包含操作不包含属性。

  • 接口不能实例化为对象。

  • 一个类可以实现一个或多个接口，一个接口可以被多个类实现。

  • 如果类实现了接口，但未实现该接口中的所有操作，那么此类必须声明为抽象的。

关联关系(Association)

  • 当类之间在概念上有实例连接关系时，这种类关系叫做关联。

  • 关联关系表明了类的对象与另一个类的对象之间的链接。

  • 例如：

(1)一个人为一家公司工作。

(2)一个人可持有多家公司的股票。

(3)每个国家都有一个首都。

(4)飞行员驾驶飞机， 飞机被某一航空公司所拥有。

关联关系——相关内容

①关联名(name)

②导航性( navigation)

③角色—关联端名(role)

④多重性(multiplicity)

⑤自反关联( self- association )

⑥限定关联( qualified association )

⑦关联上的约束( constraint on association )

⑧派生关联(derived- association )

⑨关联类( association class )

①关联名(name)

  • 关联名来描述关联的作用，清晰而简洁地说明对象间关系。使用一个动词或动词短语来命名关联。

注意：

  • 关联名并不是必需的。如果模型是无歧义的，关联名可不写。

  • 如果关联两端都明确了角色(关联端名)。**关联名**可以不写。

  • 特例情况：当模型中在相同的类中有 多重关联(相同类之间的多种不同关联关系)，就必须用关联名或角色(关联端名)来消除歧义。

②导航性( navigation)

  • 关联具有方向性(导航性)，可以是单向的，也可以是双向的。

  • 关联用箭头来表示导航性，由源类指向目标类，表示源类的一个成员是目标类的对象。

  • 单向关联(uni-directional association)：关联是单向的，从一个源类的对象可以找到目标类的对象，反过来却不可。

  • 双向关联(bi-directional)： 关联是双向的，如果一个类和另一个类是双向关联，那么这两个类的对象都互相找到对方。

③角色—关联端名(role)

  • 当一个类和另一个类发生关联时，每个类通常在关联中扮演着某种角色。角色写在关联的两端，所以也叫关联端名。

  • 角色是关联中一个类对另一个类所表现出来的职责。

  • 例如: Person类 和Company类都参与了关联WorksFor。对于公司来说，人是雇员(employee) ，对于人来说，公司是雇主(employer)。

注意：

  • 角色是关联本身的属性，而不是类的组成部分，因为一个类可以在不同的关联中扮演不同的角色。

  • 由于角色名称和关联名称都被用来描述关系的目的，所以角色名称可以代替关联名称，或者两者同时使用。

  • 与关联名称不同，角色名可以生成代码。(使用Rose工具可以直接生成代码)

角色(关联端名)也不是必须，但存在以下特例情况：

(1)当模型中在相同的类中有多重关联(相同类之间的多种不同关联关系)，就必须用关联名或角色(关联端名)来消除歧义。

(2)对于自反关联或自身关联(同一类的两个对象之间的关联)，关联端名(角色)是必须的。

④多重性(multiplicity)

  • 关联的多重性指某个类有多少个对象可以和另一个类的单个对象关联。

  • 表示多重性的方法是在参与关联的类附近的关联线上注明多重性数值。可以表达一一个取值范围、特定值、无限定的范围或一组离散值。

  • UML使用星号(*)来代表许多(more)和多个(many)。

  • 在一种语境中，两点代表or(或)关系

例如：“1..*代表一个或者多个。

  • 在另一种语境中，or关系用逗号来表示

例如：“5，10”代表5或者10。

类图与某一时刻的对象图
关联上的多重性可以对应到类属性代码上

多重性为1—-对应属性中的单对象

多重性为n—-对应属性中的对象集合(对象数组)

⑤自反关联( self- association )

  • 一个类可能与它自己发生关联,这样的关联被称为自身关联。(同一个类的不同对象之间的

关系)

  • 当一个类的对象可以充当多种角色时，自身关联就可能发生。

  • CarOccupant(车.上的人)既可能是一个司机(driver)也可能是一个乘客(passenger)。

注意：自反关联也具有导航性(方向性)

⑥限定关联( qualified association )

  • 带有限定符(qualifier)的关联称为限定关联(受限关联)。

  • 限定符常用于一对多或多对多关联。

  • 限定符的作用是在目标对象之间进行选择，将有效的多重性从“多”降为“一”

注意：引入限定关联后发生多重性的变化。(限定符减少了在关联目标端的多重性，通常由多变为一)。

限定符作用

  • 限定符这个概念在设计软件时非常有用，如果一个应用系统需要根据关键字对-一个数据集做查询操作，则经常会用到限定关联。

  • 引入限定符的一个目的就是把多重性从n降到1或0..1，这样如果做查询操作，则返回的对象至多是一个，而不会是一个对象集。如果查询操作的结果是单个对象，则这个查询操作的效率会较高。

  • 如果使用限定符后,限定符另一端的多重性仍为n，则引入这个限定符的作用就不是很大，因为查询结果仍然还是结果集。因此也可以根据多重性来判断一一个限定符的设计是否合理。。

⑦关联上的约束( constraint on association )

  • 关联上的多重性是集合基数上的约束，限制了与给定对象相关的对象数量。

  • 关联端上“多”侧对象常常没有明确的顺序，可以看成集合。

关联端上“多”侧对象如果需要有明确顺序可以用{ordered}进行约束，建立一个有序的对象集合。

例如：

屏幕上显示的窗口的顺序（栈）

银行窗口的排队的顺序（队列）

⑧派生关联(derived- association )

  • 派生关联(derived association)是指可以从其他关联计算推演得到的关联。

  • 派生关联的名字前需要加一一个斜杠“/”。

  • 在生成代码时，派生关联都不产生相应的代码。

例如：

一个公司有多个部门，一个职员为其中某部门工作，

则可推导该职员为该公司工作

⑨关联类( association class )

  • 和类一样，关联本身也可以有自己的属性和操作。此时，这个关联实际上是个关联类。

  • 关联类的可视化表示方式与一般的类相同，但是要用一条虚线把关联类和对应的关联线连接起来，来标识关联的一些附加信息。

泛化关系(Generalization)

泛化关系是指一般元素和特殊元素的分类关系。

泛化关系可以用于类、用例、参与者以及其他模型元素。

泛化关系描述了一种“is a kind of”的关系，因为子类的每个实例也都是父类的实例

  • 泛化关系可以传递，可以跨越多个多层。

  • 术语祖先(ancestor)和后代( descendant)指跨越多层的类的泛化。

注意：不要把子类嵌套得过深，深度嵌套的子类很难理解。

泛化的用途

(1)结构化描述对象

当使用泛化时，就是做出概念性的陈述：根据对象的相似性和差异来组织对象，并形成分类。比单独建模每个类要来的深刻。

(2)支持代码复用—-继承代码，体现继承性

把一般的公共信息放在一般元素中，处理某个具体特殊情况时只需定义该情况的个别信息，公共信息从一般元素中继承得到，增强了系统的灵活性、易维护性和可扩充性。

(3) 支持多态

  • 多态技术利用抽象类或接口定义操作，而用子类实现该操作的方法，达到单一接口，多种功能的目的。

  • 也可以通过子类操作覆盖从父类继承下来的操作来实现多态。

面向对象设计原则–里氏替换原则(Liskov)

里氏替换原则：在一个软件系统中，子类应该可以替换任何基类能够出现的地方，并且经过替换以后，代码还能正常工作。

类中的替换原则

父类出现的任何地方都可以用子类代替。

更广泛的替换原则

一般元素出现的地方都可以用特殊元素来代替。

里氏替换原则(Liskov) 用途

来判断在设计中的继承关系是否正确。

聚合关系(Aggregation)

  • 聚合关系是一种特殊类型的关联，表示整体与部分关系的关联。

  • 描述了“由…构成”、“包含”、“是…的一部分”的关系。

  • 聚合关系体现了复杂事物可以由简单事物组成的面向对象的思想。

关联关系和聚合关系的相同点和不同点

  • 聚合关系在代码上的形式和关联关系相似(即一个类的对象作为另一个类中的成员出现，作为其组成部分)

  • 在关联关系上体现不出整体和部分的关系。

  • 在聚合关系上一定会体现整体和部分的关系。

聚合关系的重要特性

(1)传递性( transitivity)

(2)反对称的( antisymmetric)

  • 聚合关系上可以多重性、角色和约束。

共享聚合

  • 如果聚合关系中的处于部分方的对象同时参与了多个处于整体方对象的构成，则该聚合称为共享聚合。

  • 共享聚合关系可以通过聚合的多重性反应出来。如果作为整体方的类的多重性不是1，该聚合就是共享聚合。

  • 例如：一个球队(整体方)由多个球员(部分方)组成，但是一个球员还可能参加多个球队，球队和球员的这种关系就是共享聚合。

聚合关系和组合关系

  • 可以在聚合关系上建立约束。

例如：聚集上的“or”约束，表示某个整体包含一个或另一个部分。

组合关系(Composition)

  • 组合关系是聚合关系中的一种特殊情况，是更强形式的聚合，又称强聚合,有两个额外的约束。

(1)一个部分至多属于一个整体，整体部分多重性为1

(2)部分对象的生命周期取决于组合对象的生命周期，部分类完全隶属于整体类。

  • 组合不仅控制着成员对象的行为，而且控制着成员对象的创建和析构。如果删除组合对象则要触发删除所有部分对象。

聚合和组合的两端也可以添加角色和多重性。

实现关系(Realization)

  • 一个元素完成另一个元素的操作功能。 (实现是规格说明和具体实现之间的关系。)

  • 例如：具体类实现抽象类、具体类实现接口。

例如：主动类

  • 主动对象( avtive object) :运行于自己控制的执行线程之上。

  • 主动对象的类即为主动类(用双竖线或者黑色边框表示)

依赖关系(Dependency)

  • 依赖可以视为耦合(coupling)的另一个版本，耦合是软件开发中的传统术语，意为某元素耦合或依赖于另一个元素。

  • 关联、实现和泛化都是依赖关系。但它们有更特别的语义，所以在UML中被分离出来作为独立的关系。

使用依赖—最常使用的依赖

  • 表示一种元素(客户)使用其它元素(提供者)提供的服务以实现它的行为，包括:

①使用(《use》)

②调用(《call》)

③参数《parameter》 )

④发送(《send》)

⑤实例化(《instantiate》 )

各类依赖

①使用《use》:声明使用一个模型元素需要用到已存在的另一个模型元素，这样才能实现使用者的功能(包括调用、实例化、参数、发送)。

②调用《call》:声明一个类调用其他类的操作的方法。

③参数《parameter》 :声明一个操作和它的参数之间的关系。

④发送《send》:声明信号发送者和信号接受者之间的关系。(例如遥控器和电视机)

⑤实例化《instantiate》 :声明 一个类的方法创建了另一个类的实例。

注意：与关联关系不一样的是，依赖关系本身不生成专门的实现代码。

分析类介绍

  • 在系统分析阶段将分析类分成三种类型：实体类、边界类、控制类。

  • UML中类有三种主要的版型:边界类《boundary》、控制类《control》和实体类《entity》。

引入边界类、控制类及实体类的概念有助于分析和设计人员确定系统中的类。

边界类《boundary》

  • 边界类用于描述外部参与者与系统之间的交互。

  • 识别边界类可以帮助开发人员识别出用户对界面的需求。

  • 边界类位于系统与外界的交界处，窗体、对话框、报表、以及表示通讯协议的类、直接与外部设备交互的类、直接与外部系统交互的类等都是边界类。

  • 边界类可以通过用例确定，因为参与者必须通过边界类参与用例。

  • 通过用例图可以确定需要的边界类，每个actor/user case对至少要有一个边界类。但并非每个actor/user case对都要生成唯一的边界类。例如：多个actor启动一个用例时，可以用一一个边界类与系统通信。

边界类的分类

一个系统可能会有多种边界类:

(1)用户界面类：帮助与系统用户进行通信的类，通过标准的I/O设备提供人机界面。

  • 参与者与用例之间应当建立边界类：用例可以提供给参与者完成功能的操作只能通过边界类暴露出来。

  • 例如：参与者通过一组网页、一组Windows窗口、或者字符终端或鼠标来使用用例功能。都可以称为用例的边界类。

(2)系统接口类：帮助与其他系统进行通信的类，系统接口对象隐藏如何与外部接口通信的细节。

与外部系统通信的边界类负责管理与外部系统的对话，它为正在构建的系统提供与该外部系统的接口。

例如：在ATM系统中，提款必须通过ATM网络得到验证,然后该网络再通过银行会计系统对提款进行验证。我们可以确定一个ATM网络接口，让其对象来提供与ATM网络之间的通信。

(3)设备接口类：为用来监测外部事件的(硬件)设备(如传感器、Timer)提供接口的类。

★从架构角度上来说，边界类主要位于展现层(用户界面层)

实体类《entity》

  • 实体类是用于对必须存储的信息和相关行为建模的类。

  • 实体类是持久的，是永久保存的信息(需要保存到数据库或文件中)。

  • 通常，每个实体类在数据库中有相应的表，实体类中的属性对应数据库中表的字段。但这并不意味着，实体类和数据库中的表是一一对应的。有可能是一个实体类对应多个表，也可能是多个实体类对应一个表。( 与数据库设计模式有关)。

  • 例如：银行系统中实体类的典型示例是账户和客户。

  • 实体对象可以出现在多个用例中。

  • 实体类可以通过事件流和交互图发现。

  • 有时Actor也可作为实体类。

★从架构角度上来说，实体类主要位于数据持久层(数据层)

控制类《control》

  • 控制类用于对一个或几个用例所特有的控制行为进行建模。

  • 控制类对象通常被看成一个乐队的指挥，它指挥(控制)参与use case的其它对象的行为，通知对象什么时候执行以及执行什么，完成功能实现的总体逻辑流程。

  • 其它类并不向控制类发送很多消息，而是控制类向其它类对象发出很多消息。

  • 控制类将用例的特有行为进行封装。

  • 一般每个用例都有一个控制类，控制用例中的事件顺序，控制类也可以在多个用例间共用。

  • 在UML的定义中，控制类主要起到协调对象的作用，例如：从边界类通过控制类访问实体类，或者实体类通过控制类访问另一个实体类。

  • 但不必强行使用控制类，例如：边界类也可以直接访问实体类。

★从架构角度上来说，控制类主要位于业务逻辑层。
